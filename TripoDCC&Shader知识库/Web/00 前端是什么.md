## 零、先破除三个误解（零基础必读）

你脑子里对"网页"可能是空白或有误解，先校准：

1. **网页不是"软件"，它就是几个文本文件。**
   一个网页，本质是服务器上放着的**几份纯文本文件**：`.html`、`.css`、`.js`。它们跟你用记事本能打开的 `.txt` 没有本质区别，只是里面写的内容遵循特定语法。浏览器把这些文本**下载**下来，再**翻译**成你看到的画面。
   > 类比：就像你的引擎加载一个**场景描述文件**（`.scene`/`.prefab`）——文件本身是死的文本，引擎读它、在内存里建出物体、渲染上屏。网页同理，浏览器就是那个"引擎"。

2. **浏览器才是真正运行网页的"引擎"。**
   Chrome / Edge / Safari 这些浏览器，是一个**通用运行时**——你可以把它想成一个"能播放任意网页的播放器"，就像 Unity 播放器能播放任意 `.unity3d` 包。你写网页 = 你在给这个播放器**喂内容**，而不是自己从头写一个能显示窗口、处理鼠标、绘制像素的程序（那些浏览器早替你做好了）。

3. **"前端"就是这层壳的开发工作。**
   - **前端（front-end）**：跑在**用户浏览器里**的部分——你看到、点到的一切界面。
   - **后端（back-end）**：跑在**远端服务器上**的部分——存数据、算业务、管账号，你看不见。
   - 两者通过**网络请求**对话（前端问、后端答，见 [[05 计算机网络]]）。
   > 类比：前端 = 游戏客户端（渲染、UI、输入）；后端 = 游戏服务器（存档、结算、匹配）。你现在学的是**客户端这一侧**。

**术语澄清**（这几个词平时混着用）：
| 词 | 指什么 |
| --- | --- |
| **网页（page）** | 一个 HTML 文件对应的一屏内容 |
| **网站（site）** | 一堆网页 + 资源的集合（如整个 `tripo3d.com`） |
| **网址 / URL** | 一个资源的"地址"，如 `https://tripo3d.com/index.html`（拆解见 [[05 计算机网络]]） |
| **浏览器（browser）** | 运行网页的程序 = 运行时/引擎 |

## 一、一句话定位

**前端 = 在浏览器里跑的程序。** 你之前写的是"跑在操作系统 / GPU 上的原生程序"，前端是"跑在浏览器这个运行时里的程序"。

浏览器就像 Unity 引擎：它提供了运行时（JS 引擎）、渲染系统（DOM/CSS）、资源加载、事件系统，你写代码去**驱动**它，而不用自己实现这些底层。

## 二、三大支柱（各管一件事，各写在不同文件里）

前端页面由三种语言协作，**职责分明，通常分别写在三种文件里**：

| 语言 | 文件后缀 | 引擎类比 | 管什么 | 例子（下面详解） |
| --- | --- | --- | --- | --- |
| **HTML** | `.html` | 场景里的物体层级（Hierarchy） | 页面有哪些**内容/结构**（标题、按钮、图片） | `<h1>标题</h1>` |
| **CSS** | `.css` | 材质 / Shader | 这些内容**长什么样**（颜色、大小、布局、动画） | `h1 { color: red; }` |
| **JavaScript** | `.js` | C# 脚本 / 游戏逻辑 | 页面的**行为**（点击、请求数据、动态更新） | `button.onclick = ...` |

**它们怎么拼成一个能看的页面**——看这个"最小网页"，一个 `.html` 文件把三者串起来：

```html
<!DOCTYPE html>              <!-- 声明：这是一个 HTML 文件 -->
<html>
  <head>                     <!-- 头部：给浏览器看的信息，不直接显示 -->
    <title>我的第一个页面</title>
    <style>                  <!-- CSS 写在这：定义"长什么样" -->
      h1 { color: #4f46e5; }
    </style>
  </head>
  <body>                     <!-- 身体：真正显示在屏幕上的内容 -->
    <h1>你好，前端</h1>       <!-- HTML：一个一级标题 -->
    <button>点我</button>    <!-- HTML：一个按钮 -->
    <script>                 <!-- JS 写在这：定义"行为" -->
      document.querySelector('button').onclick = () => alert('被点了');
    </script>
  </body>
</html>
```

把这段存成 `index.html`、双击用浏览器打开，你就会看到一个紫色标题 + 一个能弹窗的按钮。**这就是一个完整网页的全部**——不需要编译、不需要安装，浏览器直接读文本出画面。

- HTML/CSS 的每个符号具体怎么读、怎么写 → [[00.3 CSS上手]]（先讲这个，因为最直观）
- HTML 文件的完整骨架、常见标签 → [[01 HTML与CSS]]
- JS 的行为逻辑 → [[02.0 JavaScript上手]] → [[02 JavaScript核心]]
- **TypeScript** = 给 JavaScript 加了静态类型的版本，就像从"动态语言"升级到"有类型检查"。你有 C#/C++ 的静态类型背景会非常适应，见 [[03 TypeScript]]。公司项目基本都用 TS。

## 三、一个页面是怎么"跑起来"的

从你熟悉的角度理解这条链路（网络细节见 [[05 计算机网络]]，渲染细节见 [[04 浏览器原理]]）：

```
用户在地址栏输入网址
  → 浏览器通过网络，从服务器下载 HTML 文件  （相当于加载场景描述文件）
  → 解析 HTML，建立 DOM 树                （相当于把场景文件读成内存里的节点树）
  → 发现 <link>/<style>，下载并解析 CSS，算出每个节点长什么样（相当于给节点挂材质）
  → 排版(Layout) + 绘制(Paint) + 合成(Composite) 到屏幕（相当于一帧渲染）
  → 下载并执行 JS，绑定事件、之后动态修改 DOM（相当于运行时脚本驱动场景变化）
```

**DOM（Document Object Model）到底是什么**：浏览器把 HTML 文本解析后，在内存里建的一棵**活的节点树**。HTML 文件是"死的文本"，DOM 是它"活起来"的运行时版本——JS 通过操作 DOM 来改变页面。
> 这跟你在 [[Tripo Orbit]] 里操作场景节点树、标脏重绘是**同一种思想**（见 [[08 Web3D与AI]] 的脏节点）：文件 → 内存树 → 改树 → 重绘。

**两个关键差异（务必记住，跟原生开发很不一样）**：
- 浏览器里的 JS 是**单线程**的（见 [[04 浏览器原理]] 事件循环）。不像 C++ 可以随便开线程，前端重计算要靠 **Web Worker** 或拆分任务，否则界面会卡死（做 Web 端 shader 的 CPU 预处理时尤其要注意）。
- 前端没有 `main()` 那种"一次跑完"的入口，而是**事件驱动**：加载完就静静等着，用户点一下 / 数据回来了，才触发对应回调。类似 Unity 的 `Update` / 事件回调模型。

## 四、你的业务在前端的哪个位置

结合 [[任务需求]] 里你做的方向：
- **Web 端 shader**（ts/js + WebGL + GLSL）：这是"前端 + 图形"的交叉，也是你的主战场。前端负责搭页面、拿到 `canvas`（网页里的一块"可画像素的画布"）、把数据传进去；着色器那部分你已经会了——你写过 URP 下的 HLSL（GGX、次表面、Kajiya-Kay）和软光栅，迁到 **GLSL ES** 主要是语法和精度限定的差异（见 [[09 前端接WebGL]]、[[08 Web3D与AI]]）。前端要补的是**怎么把它挂到网页里、把数据喂进去**。

## 速记

- 网页 = 服务器上的几份文本文件（`.html`/`.css`/`.js`），浏览器下载后翻译成画面。
- 浏览器 = 通用运行时/引擎；前端 = 给它喂内容的开发工作（客户端侧）；后端 = 远端服务器（服务器侧）。
- HTML 管结构、CSS 管样式、JS 管行为、TS 给 JS 加类型；三者常分写在三种文件，一个 `.html` 可把它们串起来。
- DOM = HTML 在内存里"活起来"的节点树，JS 改 DOM 来更新画面，思想同你的场景节点树 + 标脏重绘。
- 单线程 + 事件驱动，是它和原生开发最大的不同。
