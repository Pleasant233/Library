# 05 计算机网络

> 配套 [[面试题库]] 的「计算机网络」部分。前端只需抓住「和性能相关」的部分。

## 一、CDN 工作原理

**CDN（内容分发网络）**：把资源缓存到全球各地的**边缘节点**，让用户从**最近的节点**取，而不是每次都回源站。

核心概念：
- **边缘节点**：分布在各地的缓存服务器。
- **就近访问**：靠 DNS 调度把用户导向最近节点，降低延迟。
- **回源**：边缘节点没有缓存时，去源站取一份并缓存。
- **缓存命中率**：命中越高，回源越少，越快、源站压力越小。

**给全球应用设计缓存策略**：
| 资源类型 | 策略 |
| --- | --- |
| 静态资源（JS/CSS/图片/字体） | 文件名带**哈希指纹** + **长 TTL**（一年），内容变则文件名变 |
| HTML 入口 | **短 TTL 或不缓存**，保证拿到最新资源引用 |
| API | 一般不走 CDN 缓存（动态数据），或用极短 TTL / `stale-while-revalidate` |

**灰度发布 / 回滚**：靠改文件指纹和 HTML 引用，出问题回退到旧文件名即可（呼应 [[04 浏览器原理]] 缓存策略）。

## 二、HTTP/1.1 vs 2 vs 3

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
| --- | --- | --- | --- |
| 底层 | TCP | TCP | **QUIC（基于 UDP）** |
| 连接 | 多条 TCP 连接 | **单连接多路复用** | 单连接多路复用 |
| 队头阻塞 | 有（应用层） | 应用层解决，但**TCP 层仍有** | **彻底解决**（QUIC 独立流） |
| 头部压缩 | 无 | **HPACK** | QPACK |

**逐条理解**：
- **HTTP/1.1**：一个连接一次只能处理一个请求，浏览器只能开多条连接（一般 6 条）并发 → 存在**队头阻塞**（前一个响应堵住后面）。
- **HTTP/2**：一条 TCP 连接上用「流」多路复用，多个请求并行；`HPACK` 压缩重复的请求头。但因为底层还是 TCP，**丢包时 TCP 队头阻塞**依然拖慢所有流。
- **HTTP/3**：换成 **QUIC（跑在 UDP 上）**，每条流独立，一条流丢包不影响其他流，**彻底解决队头阻塞**；连接迁移更快（切 WiFi/4G 不断连）。

**对前端加载策略的影响**：
- 在 HTTP/2/3 下，**传统「合并」手段收益变小甚至有害**：
  - 雪碧图、把 JS/CSS 强行打成一个大包 → **不再必要**，因为多路复用下小文件并发没成本。
  - 反而拆成小文件更利于**缓存粒度**（改一个模块只失效一个文件）和按需加载。
- **Server Push** 现在多数不推荐（易推送浪费、已被 `<link rel="preload">` / 103 Early Hints 替代）。
- 仍要做的：压缩、`preload`/`prefetch` 关键资源、合理的缓存头。

## 面试速答卡

- **CDN 为什么快**？→ 边缘节点就近访问 + 缓存命中减少回源。
- **静态资源缓存策略**？→ 哈希指纹 + 长 TTL；HTML 短缓存。
- **HTTP/2 核心改进**？→ 单连接多路复用 + HPACK 头部压缩。
- **HTTP/3 解决了什么**？→ 基于 QUIC/UDP，解决 TCP 队头阻塞。
- **HTTP/2 下还要雪碧图吗**？→ 不需要，多路复用让小文件并发无成本。
