## 零、网络地基：一次访问到底发生了什么

### 0.1 最核心的模型：客户端问、服务器答（请求-响应）

整个 Web 网络，99% 是这一个模型：

```
你的浏览器（客户端 client）  ──── 请求(request) ───►  远端服务器(server)
                            ◄─── 响应(response) ───
```

- **客户端（client）**：发起请求的一方 = 你的浏览器 / App。
- **服务器（server）**：一台一直开机、等着被请求的电脑，收到请求就返回数据。
- 你每打开一个网页、点一个按钮拉数据，本质都是**客户端发一个请求，服务器回一个响应**。

> 类比：像你游戏客户端向游戏服务器请求"给我这个玩家的存档"，服务器查完回给你。前端的每次"加载""提交"都是这种一问一答。**注意方向**：默认总是客户端主动问、服务器被动答；服务器不能主动推（想让服务器主动推，得用 WebSocket，见 [[10 前端接WebSocket]]）。

### 0.2 怎么找到服务器：IP、域名、DNS

服务器在互联网上靠 **IP 地址**标识（像 `142.250.72.14`，相当于电脑的"门牌号"）。但 IP 难记，所以有了**域名（domain）**——`tripo3d.com` 这种好记的名字。

两者的翻译由 **DNS（Domain Name System，域名系统）** 完成：

```
你输入 tripo3d.com
  → 浏览器问 DNS："tripo3d.com 的 IP 是多少？"
  → DNS 答："142.250.72.14"
  → 浏览器拿 IP 去连那台服务器
```

> 类比：DNS = 互联网的"通讯录 / 电话簿"，把人记得住的名字翻译成机器用的号码。

### 0.3 URL：一个网址的每一段是什么

你在地址栏输入的"网址"，专业叫 **URL（统一资源定位符）**。拆解一个完整的：

```
https :// tripo3d.com : 443 / models/cat.glb ? id=99&fmt=glb # section2
└─①─┘    └───②────┘  └③┘ └────④─────┘   └────⑤────┘  └──⑥──┘
```

| 段 | 名称 | 是什么 |
| --- | --- | --- |
| ① `https` | 协议 | 用什么规则通信（见 0.5）。常见 `http`/`https`/`ws` |
| ② `tripo3d.com` | 域名/主机 | 找哪台服务器（经 DNS 翻成 IP） |
| ③ `443` | 端口 | 服务器上哪个"服务窗口"，http 默认 80、https 默认 443，通常省略 |
| ④ `/models/cat.glb` | 路径 | 要那台服务器上的哪个资源 |
| ⑤ `?id=99&fmt=glb` | 查询参数 | 附带的参数，`?` 起头、`&` 分隔多个 `键=值` |
| ⑥ `#section2` | 锚点 | 页面内定位（不发给服务器，浏览器自己用） |

> 端口的类比：一台服务器（一个 IP）像一栋楼，端口就是楼里不同的"门牌/窗口"——80 号窗口办网页、443 号办加密网页。

### 0.4 HTTP：请求和响应里到底装了什么

**HTTP（超文本传输协议）** 是客户端和服务器对话用的"语言格式"。一次请求/响应各由三部分组成：

**请求（浏览器发出）**：
```
GET /models/cat.glb HTTP/1.1        ← 请求行：方法 + 路径 + 版本
Host: tripo3d.com                   ← 请求头(headers)：一堆 键:值 元信息
Authorization: Bearer sk-xxx        ←   （比如带上身份令牌）
                                    ← 空行后是请求体(body)，GET 通常没有
```

**响应（服务器返回）**：
```
HTTP/1.1 200 OK                     ← 状态行：版本 + 状态码 + 说明
Content-Type: model/gltf-binary     ← 响应头：告诉你 body 是什么类型
                                    ← 空行
<二进制模型数据...>                  ← 响应体(body)：真正要的内容
```

**请求方法（method）**——你想对资源做什么：
| 方法 | 含义 | 类比 |
| --- | --- | --- |
| `GET` | 取数据（只读） | 查询 |
| `POST` | 提交/新建数据 | 新增 |
| `PUT` / `PATCH` | 更新数据 | 修改 |
| `DELETE` | 删除数据 | 删除 |

**状态码（status code）**——服务器告诉你结果，按首位分五类：
| 段 | 含义 | 常见 |
| --- | --- | --- |
| 2xx | 成功 | `200 OK` |
| 3xx | 重定向（去别处拿） | `301 永久` `304 用缓存` |
| 4xx | **你（客户端）错了** | `401 没登录` `403 没权限` `404 找不到` |
| 5xx | **服务器错了** | `500 服务器内部错误` |

> 眼熟吗？你之前调 Codex 代理时报的 `401 Unauthorized 无效令牌`、`400 Invalid model name`，就是 4xx——服务器说"是你请求有问题"。这套状态码你以后天天见。

**请求头 / 响应头（headers）**：一堆 `键: 值` 的元信息，携带身份令牌、内容类型、缓存策略等（比如你 Codex 里那个 `Authorization: Bearer sk-...` 就是请求头）。

### 0.5 HTTP vs HTTPS：加不加密

- **HTTP**：明文传输，中间人能偷看/篡改。
- **HTTPS**：= HTTP + **TLS 加密**。数据加密传输，还能验证"对面确实是 tripo3d.com 而不是假冒的"。地址栏那把小锁就是它。**现在线上一律用 HTTPS。**

### 0.6 底层一句话：TCP 与 UDP

HTTP 的数据最终靠下面这两种"传输方式"送：
- **TCP**：可靠、按顺序、会重传丢的包——像"挂号信，保证送到"。网页、HTTP/1.1、HTTP/2 都用它。
- **UDP**：快但不保证送达/顺序——像"平信，图快不图稳"。视频通话、HTTP/3（QUIC）用它。

第 0 节到此。**你现在能看懂"一次网络请求"的全貌了**，下面是前端最关心的性能优化层。

## 一、CDN 工作原理

**CDN（内容分发网络）**：把资源缓存到全球各地的**边缘节点**，让用户从**最近的节点**取，而不是每次都回源站。

核心概念：
- **边缘节点**：分布在各地的缓存服务器。
- **就近访问**：靠 DNS 调度（0.2 讲的那个）把用户导向最近节点，降低延迟。
- **回源**：边缘节点没有缓存时，去源站取一份并缓存。
- **缓存命中率**：命中越高，回源越少，越快、源站压力越小。

**给全球应用设计缓存策略**：
| 资源类型 | 策略 |
| --- | --- |
| 静态资源（JS/CSS/图片/字体） | 文件名带**哈希指纹** + **长 TTL**（一年），内容变则文件名变 |
| HTML 入口 | **短 TTL 或不缓存**，保证拿到最新资源引用 |
| API | 一般不走 CDN 缓存（动态数据），或用极短 TTL / `stale-while-revalidate` |

**灰度发布 / 回滚**：靠改文件指纹和 HTML 引用，出问题回退到旧文件名即可（呼应 [[04 浏览器原理]] 缓存策略）。

## 二、HTTP/1.1 vs 2 vs 3

（承接 0.4：这是同一个 HTTP 协议的三代演进，主要解决"怎么更快地并发传多个资源"。）

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
| --- | --- | --- | --- |
| 底层 | TCP | TCP | **QUIC（基于 UDP）** |
| 连接 | 多条 TCP 连接 | **单连接多路复用** | 单连接多路复用 |
| 队头阻塞 | 有（应用层） | 应用层解决，但**TCP 层仍有** | **彻底解决**（QUIC 独立流） |
| 头部压缩 | 无 | **HPACK** | QPACK |

**逐条理解**：
- **HTTP/1.1**：一个连接一次只能处理一个请求，浏览器只能开多条连接（一般 6 条）并发 → 存在**队头阻塞**（前一个响应堵住后面）。
- **HTTP/2**：一条 TCP 连接上用"流"多路复用，多个请求并行；`HPACK` 压缩重复的请求头。但因为底层还是 TCP，**丢包时 TCP 队头阻塞**依然拖慢所有流。
- **HTTP/3**：换成 **QUIC（跑在 UDP 上）**，每条流独立，一条流丢包不影响。

**对前端加载策略的影响**：
- 在 HTTP/2/3 下，**传统"合并"手段收益变小甚至有害**：
  - 雪碧图、把 JS/CSS 强行打成一个大包 → **不再必要**，因为多路复用下小文件并发没成本。
  - 反而拆成小文件更利于**缓存粒度**（改一个模块只失效一个文件）和按需加载。
- **Server Push** 现在多数不推荐（易推送浪费、已被 `<link rel="preload">` / 103 Early Hints 替代）。
- 仍要做的：压缩、`preload`/`prefetch` 关键资源、合理的缓存头。

## 面试速答卡

- **一次请求的过程**？→ DNS 解析域名→IP → 建连接(TCP/TLS) → 发 HTTP 请求 → 服务器返响应 → 浏览器渲染。
- **常见状态码**？→ 200 成功、301/304 重定向/缓存、401/403/404 客户端错、500 服务器错。
- **GET vs POST**？→ GET 取数据无副作用可缓存、POST 提交数据带 body。
- **CDN 为什么快**？→ 边缘节点就近访问 + 缓存命中减少回源。
- **静态资源缓存策略**？→ 哈希指纹 + 长 TTL；HTML 短缓存。
- **HTTP/2 核心改进**？→ 单连接多路复用 + HPACK 头部压缩。
- **HTTP/3 解决了什么**？→ 基于 QUIC/UDP，解决 TCP 队头阻塞。
- **HTTP/2 下还要雪碧图吗**？→ 不需要，多路复用让小文件并发无成本。

## 速记

- 核心模型：客户端**请求**、服务器**响应**（一问一答，服务器不主动推 → 要推用 WebSocket）。
- 找服务器：域名 --DNS--> IP；URL = 协议://域名:端口/路径?参数#锚点。
- HTTP：请求(方法+头+体) / 响应(状态码+头+体)；状态码 2 成 3 转 4 你错 5 我错。
- HTTPS = HTTP + 加密；底层 TCP 可靠、UDP 快。
- 性能层：CDN 就近缓存；HTTP/1.1→2→3 一路解决并发与队头阻塞。
