# 00.2 与同事对接

> 你 7.9 要提 PR 发布 Unity DCC 应用（见 [[任务需求]]），这篇讲**协作流程**和**对接时的黑话**，让你能顺畅地和前端同事合作。

## 一、Git 协作流程（最常用）

你可能单人开发过 Git，但团队协作有固定套路。一次改动的标准流程：

```bash
git checkout main
git pull                          # 先同步最新代码
git checkout -b feat/xxx          # 从 main 切出一个功能分支
# ... 改代码 ...
git add .
git commit -m "feat: 支持动画导入"  # 提交（见下方提交规范）
git push -u origin feat/xxx       # 推到远程
# 然后去 GitHub/GitLab 上开 PR
```

**核心原则**：
- **不直接往 `main` 推**。永远开分支 → 提 PR → review 后合并。
- 分支名有约定：`feat/` 新功能、`fix/` 修 bug、`docs/` 文档、`refactor/` 重构。
- 经常 `git pull` 同步主干，避免冲突堆积。

## 二、提交信息规范（Conventional Commits）

团队常要求这种格式，工具（和 [[00.1 环境与工具链]] 的 Git Hooks）会校验：

```
<类型>: <简短描述>

feat: 新功能
fix: 修复 bug
docs: 只改文档
style: 格式（不影响逻辑）
refactor: 重构（非功能非修 bug）
perf: 性能优化
test: 测试
chore: 构建/工具/杂项
```
例：`fix: 修复材质球名称多了 BaseColor 的问题`（正好对应你 7.8 做的事）。

## 三、PR（Pull Request）是什么

**PR = 「请把我这个分支的改动合并进主干」的申请**，是团队 review 代码的地方。

一个好的 PR：
- **标题**清楚：`feat: Unity DCC 1.0.1 发布`。
- **描述**写：改了什么、为什么、怎么测的、有没有风险点、相关截图。
- **改动聚焦**：一个 PR 只做一件事，别把十个不相关的改动塞一起（reviewer 会崩溃）。
- 关联 issue / 需求。

**Code Review 流程**：
1. 你开 PR，指定 reviewer（通常是带你的同事）。
2. 同事逐行评论，提出修改意见。
3. 你改完 push，评论会跟着更新。
4. 通过（Approve）后才能合并（Merge）。
5. **CI**（自动化流水线）会自动跑 lint / 类型检查 / 测试，红了要修好才能合。

> 心态：Review 意见是对**代码**不是对**人**。有不懂的直接问，这是最快的学习方式。

## 五、遇到问题怎么高效提问

新人最重要的软技能。别只说「不行了」，按这个模板问同事：
1. **我想做什么**（目标）
2. **我做了什么**（步骤 / 贴关键代码）
3. **期望什么、实际什么**（现象 + 报错原文，别截半张图）
4. **我查过什么**（体现你先自己试过）

> 报错信息**原文复制**给同事/搜索，比描述「它坏了」有用一百倍。

## 六、对你当前任务的具体建议

- 提 PR 前**本地先跑一遍** `pnpm build` / lint，别让 CI 替你发现低级错误。
- Unity Bridge 网页侧和插件侧对接，本质是**约定好数据格式**（分片传输协议见 [[Unity Bridge]]），改协议要两边同步，一定在 PR 里说清。
- 发布 1.0.1 记得更新版本号、写 changelog、打 tag。

## 速记

- 开分支 → commit（`feat:`/`fix:` 规范）→ push → 开 PR → review → merge。
- 不直接推 main；一个 PR 只做一件事。
- Review 是对代码不对人，不懂就问。
- 提问带：目标 / 步骤 / 现象报错原文 / 已尝试。
