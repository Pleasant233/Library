# 11 实战项目路线（AI 协作版）

> 承接 [[前端知识地图]]。这篇是**动手实践**的主线：8 级项目，从"一个静态页"爬到"Web 端模型 + shader 预览器 + 桥接插件"——**终点就是你 [[任务需求]] 里的真实业务**。
> **新规范（重要）**：代码**全部由 AI 写**。你的角色不是"打字员"，而是**读懂 / 验证 / 定架构**的那个人。所以每一级都明确列出"这一级你必须自己能读懂和检验的范式"——这才是你要学的，语法只是用来**看懂 AI 产出、判断它对不对**。
> 方法论（为什么这样排）见 [[学习路径设计方法论]]。

---

## 怎么用这份路线（每一级的固定结构）

每个项目都按同一套模板推进，照做即可：

1. **🎯 产物**：做出什么，一句话。
2. **🧠 必须自己会的范式**：这一级引入的**新概念**（只 1~2 个）。做完你要能**脱离 AI 讲清楚**它是什么、对应你已有的什么经验。
3. **🤖 怎么驱动 AI**：给 AI 的指令怎么下（prompt 套路），以及**别让 AI 替你做的部分**。
4. **✅ 验收（自测，不盲信 AI）**：几个你**亲手验证**的动作。能过 = 真懂；过不了 = 回头补对应文档。
5. **📎 对应文档**：卡壳时回哪篇。

> **核心心法**：AI 写完 ≠ 你学会。判据永远是"**我能不能改一个地方、并预测它的结果**"。改完跑一下，结果和预测一致 = 你真懂了这个范式。

## 前置：搭好"人机协作"的最小环境

在开始前用 AI 帮你把脚手架起好（细节见 [[00.1 环境与工具链]]）：

```bash
# 让 AI 执行/解释这几步，你要看懂每步在干嘛
pnpm create vite my-lab --template vanilla-ts   # 起一个 Vite 项目
cd my-lab && pnpm install && pnpm dev            # 装依赖、开发服务器
```

- **你必须理解的**：Vite = 开发服务器 + 打包器（改代码浏览器自动刷新，见 HMR）；`pnpm` = 包管理器（装第三方库）；`package.json` = 项目清单。
- **给 AI 的开场指令模板**：
  > "我在学前端，用 Vite + TS。请帮我实现 X。**要求：① 代码尽量简单直白；② 每个我可能不懂的语法加一行注释；③ 做完告诉我‘我应该改哪一行来验证我真的理解了’。**"

## L1 · 静态个人卡片页（纯 HTML/CSS）

- **🎯 产物**：一张居中的"个人名片"卡片——头像、名字、简介、几个标签，手机/电脑都不变形。
- **🧠 必须自己会的范式**：
  - HTML 是**元素树**（谁套谁）；`class` 是贴给元素、供 CSS 选中的牌子。
  - **盒模型**（content/padding/border/margin）+ `box-sizing: border-box`。
  - **Flex 居中** 和一个**响应式断点**。
- **🤖 怎么驱动 AI**：让它生成整页，但**要求它用 Flex 而不是绝对定位**，并让它解释"卡片为什么能水平垂直居中"。
- **✅ 验收（自测）**：
  1. 不看 AI，自己把卡片内边距从 `16px` 改成 `32px`，**预测**卡片会变大还是内容变挤，再刷新验证。
  2. 把 `box-sizing` 那行删掉，观察布局怎么错位，说清为什么（呼应盒模型那个坑）。
  3. 把浏览器窗口拉窄到手机宽度，卡片应从"占屏一半"变"占满宽"。找到是哪条 `@media` 在起作用。
- **📎 对应文档**：[[00.3 CSS上手]]、[[01 HTML与CSS]]。

## L2 · 交互计数器 / To-Do（原生 JS）

- **🎯 产物**：一个能"输入待办 → 回车添加 → 点击删除 → 数量实时更新"的小应用。
- **🧠 必须自己会的范式**：
  - **DOM 操作**：JS 通过 `document.querySelector` 拿到元素、`addEventListener` 绑事件、`createElement`/`append`/`remove` 增删节点——这就是 [[00 前端是什么]] 说的"改 DOM 树来更新画面"。
  - **事件驱动**：没有 `main()`，是"用户点了 → 回调触发"。对应你熟悉的 Unity 事件回调。
  - **状态在内存里**：一个数组存待办，界面是这个数组的"渲染结果"。
- **🤖 怎么驱动 AI**：让它先用**最朴素的 DOM 操作**实现（别上框架），因为你要先亲眼看见"手动改 DOM"有多啰嗦——这样你才懂 L4 框架解决的是什么痛。
- **✅ 验收（自测）**：
  1. 说出点"添加"按钮后，从事件到界面更新，**中间发生了哪几步**。
  2. 自己加一个"全部清空"按钮（先想好要动哪个数组、再让 AI 帮写，最后你验证）。
  3. 故意制造一个 bug（比如删除时下标错位），用浏览器 **DevTools 断点**找到它。
- **📎 对应文档**：[[02.0 JavaScript上手]]、[[02 JavaScript核心]]。

## L3 · 拉数据的信息卡（fetch + 异步）

- **🎯 产物**：输入一个 GitHub 用户名 → 请求接口 → 显示头像/仓库数/简介；带**加载中**和**出错**两种状态。
- **🧠 必须自己会的范式**：
  - **请求-响应**：`fetch(url)` 发 HTTP 请求，`await` 等响应回来，`response.json()` 解析数据——**这就是 [[05 计算机网络]] 第 0 节讲的一问一答**，现在你亲手发一次。
  - **异步 async/await**：网络是"发出去、等一会儿才回"，代码不能干等，用 `await` 表达"等它回来再继续"。
  - **三态渲染**：加载中 / 成功 / 失败，每种都要有 UI（新手最容易只写成功态）。
- **🤖 怎么驱动 AI**：让它实现，并**要求把 loading / error 两种状态都做出来**，还要在 Network 面板能看到那条请求。
- **✅ 验收（自测）**：
  1. 打开 DevTools 的 **Network** 面板，找到那条请求，说出它的**方法、状态码、返回的 JSON** 长什么样。
  2. 把用户名改成一个不存在的 → 应显示错误态而不是白屏。对应哪个**状态码**（404）？
  3. 把网络调成"Slow 3G"，观察 loading 态。
- **📎 对应文档**：[[05 计算机网络]]、[[02 JavaScript核心]]（异步）。

## L4 · 用 Vue 重写 To-Do（组件化 + 响应式）

- **🎯 产物**：把 L2 的 To-Do 用 **Vue** 重写一遍，功能一样但代码结构完全不同。
- **🧠 必须自己会的范式**（**这一级是思维转折点**）：
  - **声明式 vs 命令式**：L2 你**手动**改 DOM（命令式）；Vue 里你只改**数据**，界面自动跟着变（声明式）。这是前端最大的范式跃迁。
  - **响应式（reactive）**：数据变 → 视图自动更新，不用再 `querySelector` 手动操作 DOM。类比：像你在引擎里改一个绑定的属性，UI 自动刷新。
  - **组件（component）**：把"一条待办"封装成可复用积木，通过 **props** 接收数据。
- **🤖 怎么驱动 AI**：让它用 **Vue 3 `<script setup>` + `ref`** 实现，并**逐行对比 L2**："同样一个删除操作，原生 DOM 版和 Vue 版分别改了什么？"
- **✅ 验收（自测）**：
  1. 说清"我只改了数组，界面为什么会自己更新"——响应式在背后做了什么。
  2. 自己把"一条待办"抽成独立组件，用 props 传数据（想好接口，再让 AI 落地）。
  3. 对比 L2 和 L4：**哪些手动 DOM 代码消失了**？消失的部分就是框架帮你做的。
- **📎 对应文档**：[[06 Vue与React]]。

## L5 · 多页应用（Vue Router + 状态管理）

- **🎯 产物**：一个有"列表页 → 详情页"的小站，点条目切换页面**不刷新**，跨页共享数据（如当前选中项）。
- **🧠 必须自己会的范式**：
  - **前端路由 / SPA**：整站只加载一次 HTML，点链接**不向服务器要新页面**，而是由 JS 换内容（单页应用 SPA）。对应 [[05 计算机网络]] 里 URL 的 `路径`/`锚点`。
  - **组件树 + 数据流**：数据往下传（props）、事件往上抛（emit），跨层共享用状态库（Pinia）。
- **🤖 怎么驱动 AI**：让它搭 Vue Router + 两个页面，并解释"点链接为什么没触发整页刷新"。
- **✅ 验收（自测）**：
  1. 在 Network 面板确认：切换页面时**没有**新的 HTML 文档请求（这就是 SPA）。
  2. 手动改地址栏 URL，能直达详情页吗？（练路由参数）
  3. 说清一份"当前选中项"数据是怎么被两个页面共享的。
- **📎 对应文档**：[[06 Vue与React]]、[[07 Nuxt与SSR]]（了解即可）。

## L6 · 图形第一步：Canvas / WebGL + 一个 fragment shader

> **从这里开始进你的主场**——图形。前面 5 级是为了"把 shader 挂进网页"打地基。

- **🎯 产物**：一块 `<canvas>`，用 WebGL 画一个全屏三角形，跑一个**你写的 fragment shader**（比如随时间流动的渐变/噪声）。
- **🧠 必须自己会的范式**：
  - **前端如何托管图形**：拿到 `canvas` → 取 `WebGLRenderingContext` → 编译 GLSL → 传 **uniform**（时间、分辨率、鼠标）→ 每帧 `requestAnimationFrame` 重绘。**着色器你已经会**（你写过 HLSL），要补的是**外面这层 JS 胶水**。
  - **GLSL ES vs HLSL**：语法/精度限定差异（`precision mediump float;` 等），见 [[09 前端接WebGL]]。
- **🤖 怎么驱动 AI**：让它写**JS 胶水部分**（建 context、编译、传 uniform、渲染循环），**GLSL 片元你自己写**——这正好是"AI 写你不熟的、你写你擅长的"分工示范。
- **✅ 验收（自测）**：
  1. 把一个 uniform（如时间流速）从 JS 传进去，改它、看画面变化——确认"JS ↔ shader 的数据通道"通了。
  2. 只改 GLSL 里的颜色公式，不动 JS，画面应随之变（证明你分得清两侧职责）。
  3. 讲清这套渲染循环和你软光栅/URP 里"每帧"的对应关系。
- **📎 对应文档**：[[09 前端接WebGL]]、[[08 Web3D与AI]]。

## L7 · glTF 模型查看器（Three.js）

- **🎯 产物**：拖入一个 `.glb` → 在网页里显示 3D 模型，能轨道旋转、缩放，可切换材质/给它套自定义 shader。**这是 [[Tripo Orbit]] 的 Web 版雏形。**
- **🧠 必须自己会的范式**：
  - **Three.js 场景图**：Scene / Camera / Mesh / Material / Light——**几乎和 Unity 的场景概念一一对应**，迁移成本低。
  - **glTF/glb 加载**：`GLTFLoader` 把模型读成场景节点（呼应你 [[任务需求]] 里 fbx/glb/usd 数据白盒）。
  - **自定义材质**：用 `ShaderMaterial` 挂你 L6 写的 GLSL。
- **🤖 怎么驱动 AI**：让它用 Three.js 搭 viewer 骨架（loader + OrbitControls + 渲染循环），你负责**材质/shader** 和**验证模型加载正确**。
- **✅ 验收（自测）**：
  1. 换一个 `.glb` 能正常显示；说出模型进来后在场景图里是什么结构。
  2. 给模型换上你的 `ShaderMaterial`，确认自定义 shader 生效。
  3. 对比 Three.js 的 Scene 和 Unity Hierarchy，列出对应关系。
- **📎 对应文档**：[[08 Web3D与AI]]、[[09 前端接WebGL]]、[[Tripo Orbit]]。

## L8 · 实时桥接（WebSocket，mock 一个 DCC Bridge）

- **🎯 产物**：网页和一个本地 mock 服务用 **WebSocket 双向通信**——网页把模型数据**分片**发过去，服务端收齐回执。**这是 [[Unity Bridge]] 网页侧的缩影。**
- **🧠 必须自己会的范式**：
  - **WebSocket vs HTTP**：HTTP 是一问一答、服务器不能主动推；WebSocket 是**长连接、双向、服务器能主动推**（呼应 [[05 计算机网络]] 0.1 那句"要推用 WebSocket"）。
  - **二进制分片协议**：大模型分块传、按帧组装（见 [[10 前端接WebSocket]]、[[Unity Bridge]] 的分片协议）。改协议要两端同步——正是你 [[00.2 与同事对接]] 里说的对接要点。
- **🤖 怎么驱动 AI**：让它写网页侧 WS 客户端 + 一个 Node mock 服务端，**你重点验证"分片—重组"正确、协议两端一致**。
- **✅ 验收（自测）**：
  1. 在 Network 的 WS 面板看到帧来回；说清一次"发送模型"被拆成了几帧、怎么重组。
  2. 故意改一端的协议字段，观察对接失败——体会"协议要两端同步"。
  3. 讲清这套和你真实 Unity Bridge 的对应。
- **📎 对应文档**：[[10 前端接WebSocket]]、[[Unity Bridge]]、[[05 计算机网络]]。

## 🏁 Capstone · Web 端模型 + Shader 预览器（合体）

把 L6+L7+L8 合成一个真东西，**等于你工作的 MVP**：

> 上传 `.glb` → Three.js 预览（L7）→ 挂自定义 GLSL 材质、实时调参（L6）→ 通过 WebSocket 把结果/数据发给（mock 的）Unity 插件（L8）。

- **这一级 AI 帮你搭架子，你做集成决策**：状态怎么组织、三块怎么解耦、性能瓶颈在哪（单线程！重预处理丢 Web Worker，见 [[00 前端是什么]]）。
- **验收**：能对一个陌生人讲清整条数据链路（上传→解析→渲染→shader→传输），并指出每一环用了前面哪一级的范式。到这一步，你已经具备了独立做 Web 图形前端的能力。

## 通用：每一级都遵守的 3 条 AI 协作纪律

1. **先预测，再运行**：改任何一行前，先口头说"我预测结果是 X"，跑完对照。对上了才算学会。
2. **AI 写不熟的，你写擅长的**：JS/框架胶水丢给 AI；shader/数据结构/架构判断自己扛。分工本身就是训练。
3. **报错原文喂回去**：卡住时把**完整报错**贴给 AI（别描述"它坏了"），这和 [[00.2 与同事对接]] 的高效提问同理。

## 速记

- 路线：静态页 → 交互JS → 拉数据 → Vue组件化 → 路由多页 → WebGL shader → Three.js模型 → WebSocket桥 →（合体）预览器。
- 每级只加 1~2 个新范式，配"自己能验证"的动作；AI 写代码，你练"读懂+改+验证"。
- 前 5 级打网页地基，后 3 级进你的图形主场，Capstone = 你的真实业务 MVP。
- 判据始终是：**能不能改一处并预测结果**。
- 方法论（怎么迁移到别的学科）→ [[学习路径设计方法论]]。
