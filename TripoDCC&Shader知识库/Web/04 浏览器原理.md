# 04 浏览器原理

> 配套 [[面试题库]] 的「浏览器知识」部分。这块是前端面试的「必考核心」。

## 一、从输入 URL 到页面渲染

分五个阶段记：

1. **找地址（DNS 解析）**：把域名 `example.com` 换成 IP。有多级缓存（浏览器 → 系统 → 路由器 → DNS 服务器）。
2. **建连接**：
   - **TCP 三次握手**建立连接。
   - HTTPS 再加 **TLS 握手**协商加密密钥。
3. **发请求收响应**：发 HTTP 请求，服务器返回 HTML。
4. **渲染**（见下一节）：解析 HTML → DOM，CSS → CSSOM，合成 Render Tree → Layout → Paint → Composite。
5. **加载子资源**：遇到 CSS/JS/图片再发请求，可能阻塞渲染。

**阻塞因素**：
- CSS 是「渲染阻塞」资源（要等 CSSOM 才能渲染）。
- `<script>` 默认「解析阻塞」——遇到就停下来下载执行。用 `defer`（等 HTML 解析完再按顺序执行）或 `async`（下载完立刻执行，不保证顺序）优化。

**首屏优化**：减少阻塞资源、压缩（gzip/brotli）、HTTP/2 多路复用、关键 CSS 内联、图片懒加载。详见 [[05 计算机网络]]。

## 二、渲染流水线（重点）

```
HTML ──解析──> DOM ─┐
                     ├─> Render Tree ─> Layout ─> Paint ─> Composite ─> 屏幕像素
CSS  ──解析──> CSSOM ┘
```

| 阶段 | 做什么 |
| --- | --- |
| **DOM** | HTML 解析成节点树 |
| **CSSOM** | CSS 解析成样式树 |
| **Render Tree** | DOM + CSSOM 合并，只含**可见**节点（`display:none` 不进来） |
| **Layout（回流/reflow）** | 计算每个节点的**几何信息**（位置、大小） |
| **Paint（重绘/repaint）** | 把节点画成**像素**（颜色、边框、阴影） |
| **Composite** | 把多个**图层**合成到一起显示 |

**回流 vs 重绘（高频）**：
- **回流**：几何变了（宽高、位置、字体大小、增删节点）→ 必须重新 Layout，**代价大**。
- **重绘**：只是外观变了（颜色、背景）→ 跳过 Layout 直接 Paint，代价较小。
- **合成**：`transform`、`opacity` 只在**合成阶段**处理，**不触发** Layout 和 Paint，最便宜——所以动画优先用它俩。

**怎么避免不必要的回流**：
- **批量读写**：别在循环里「读一下布局值、改一下样式」交替进行（会触发**强制同步布局 / layout thrashing**）。先集中读，再集中写。
- 用 `transform` 代替改 `top/left` 做动画。
- 频繁变动的元素提升为**独立合成层**（`will-change: transform`）。
- 用 `requestAnimationFrame` 把视觉更新对齐到下一帧。

## 三、浏览器缓存层次

从快到持久：

| 层次 | 说明 | 场景 |
| --- | --- | --- |
| **内存缓存** | 存在 RAM，最快，关页面就没 | 当前页面重复资源 |
| **HTTP 缓存** | 磁盘，靠响应头控制 | 静态资源（JS/CSS/图片） |
| **Service Worker** | 可编程拦截请求 | PWA、离线 |
| **localStorage / sessionStorage** | key-value，约 5MB，同步 | 小配置、token |
| **IndexedDB** | 结构化数据库，容量大，异步 | 大量结构化数据、离线数据 |

**HTTP 缓存：强缓存 vs 协商缓存（必答）**：
- **强缓存**：直接用本地，不发请求。
  - `Cache-Control: max-age=31536000`（推荐）、`Expires`（旧）。
- **协商缓存**：发请求问服务器「变了没」，没变返回 **304**，不传 body。
  - `ETag`（内容指纹）配 `If-None-Match`；`Last-Modified` 配 `If-Modified-Since`。

**「既快又不过期」的经典策略**：
- 静态资源：文件名带**哈希指纹**（`app.3f9a.js`）+ **长 `max-age`**。内容变了文件名就变，天然失效，无需协商。
- HTML：**不缓存或短缓存**（协商缓存），保证入口能拿到最新的资源引用。
- Service Worker 策略：`cache-first`（离线优先）、`network-first`（新鲜优先）、`stale-while-revalidate`（先给缓存、后台更新）。

## 四、浏览器安全模型

**同源策略（Same-Origin Policy）**：协议 + 域名 + 端口三者全相同才算同源。它限制：
- 跨源读取对方页面的 DOM。
- 跨源读取 Cookie / localStorage。
- 跨源 XHR/fetch **读响应**（请求能发出，但读不到结果）。

**CORS（跨源资源共享）**：让服务器**显式允许**跨源访问的机制。
- 简单请求：直接发，服务器返回 `Access-Control-Allow-Origin: <你的源>` 就放行。
- 复杂请求（自定义头、PUT/DELETE 等）：先发 **预检请求（OPTIONS）** 问「允许吗」，通过了才发真正请求。
- **核心**：CORS 是**服务器配置**为主，前端只是配合（比如 `credentials: 'include'`）。

**两种常见攻击**：
| | XSS | CSRF |
| --- | --- | --- |
| 原理 | 注入恶意脚本在受害者页面执行 | 借用户已登录的 Cookie，伪造请求 |
| 前端防护 | 输出转义、不用 `innerHTML` 拼不可信内容、设 **CSP** | **SameSite Cookie**、校验 token、关键操作二次确认 |

## 五、单线程、事件循环与渲染

**模型**：JS 单线程 = **调用栈（Call Stack）** + **任务队列**。
- **宏任务队列**：`setTimeout`、`setInterval`、I/O、UI 事件。
- **微任务队列**：`Promise.then`、`queueMicrotask`、`MutationObserver`。

**一轮事件循环**：
```
执行一个宏任务 → 清空所有微任务 → （可能）渲染一帧 → 下一个宏任务
```
所以微任务总是「插队」在当前宏任务后、渲染前全部执行完。这解释了 [[02 JavaScript核心]] 里那道 Promise 输出题。

**渲染时机**：一般在一轮宏任务 + 微任务清空之后。**长任务会阻塞渲染和交互**（页面卡死、点击无响应）。

**优化长任务**：
- **拆分**：把大计算切成小块，用 `setTimeout` / `MessageChannel` 分批，或 `requestIdleCallback` 在空闲时做。
- **`requestAnimationFrame`**：视觉更新放到下一帧前，避免掉帧。
- **Web Worker**：把纯计算丢到子线程，主线程不卡。
- **防抖 / 节流**：高频事件（scroll、resize、input）限制执行频率。

## 面试速答卡

- **URL 到渲染**？→ DNS → TCP/TLS → 请求响应 → 渲染流水线 → 加载子资源。
- **回流 vs 重绘**？→ 几何变=回流(贵)，外观变=重绘；transform/opacity 走合成(最便宜)。
- **强缓存 vs 协商缓存**？→ 强缓存不发请求，协商缓存问 304；静态资源哈希指纹+长 max-age。
- **同源策略限制啥**？→ DOM/Cookie/fetch 读取；CORS 靠服务器放行。
- **长任务卡顿怎么办**？→ 拆分 + rAF + Web Worker + 防抖节流。
