### 1\.5 高性能自发光描边

背景：

1\.  Mask 阶段： 假设渲染管线已经将目标物体的 Alpha 通道（或特定的渲染纹理）渲染为 \`uMaskTexture\`（物体区域为 1\.0，背景为 0\.0）。
2\.  Shader 实现（GLSL）：
    \* 在片元着色器中实现 Jump Flood Algorithm \(JFA\) 的一个核心迭代步骤，或者使用 双向降采样高斯模糊 \(Dual Filtering\) 来生成扩展边缘。

## AI

### Model、LLM、Agent、RAG、Function Calling 的区别是什么？

要点：Model 是底座，LLM 是大语言模型，RAG 是检索增强，Function Calling 是让模型调用工具，Agent 是带目标、计划和工具使用能力的系统。

- Model / LLM 负责生成和推理。

- RAG 把外部知识接进来，解决“模型不知道最新信息”的问题。

- Function Calling 让模型按结构化方式调用搜索、下单、发请求等工具。

- Agent 则是把“目标 \+ 计划 \+ 工具 \+ 记忆/上下文”组合起来完成多步任务。

- 在前端里，它们通常出现在搜索问答、AI 助手和工作流面板中。



### 如何设计 AI 聊天前端的流式输出、停止生成、重新生成？

要点：是否理解 流式 UI、状态管理和用户控制权。

好的回答应该提到：token 级增量渲染、请求取消、消息状态机，以及“正在生成 / 已停止 / 可重新生成”的交互设计。

- 流式输出用 SSE、fetch stream 或 WebSocket，把 token 逐段追加到消息气泡里。

- 停止生成时要中断请求，并把当前答案状态标记成“已停止”。

- 重新生成通常保留 prompt 和历史上下文，重新发起一次请求。

- 前端要把消息分成 user、assistant、tool、error 等状态，避免 UI 乱跳。



### 什么是 Prompt Engineering？怎么写稳定的代码生成 Prompt？

要点：Prompt 不是“魔法咒语”，而是把任务、约束、输出格式和示例写清楚。

- Prompt Engineering 是通过清晰指令、约束和示例，让模型更稳定地产出目标结果。

- 写前端代码生成 prompt 时，要明确技术栈、目录结构、命名规则、边界条件和输出格式。

- 最好要求模型先输出计划，再输出代码，再输出测试点。

- 如果任务复杂，可以拆成多轮，而不是一次把所有要求塞进去。



### 什么是 Context Engineering？如何让 AI 更懂代码库？

要点：Context Engineering 的核心是“把模型看到的内容组织好”，让它更容易做对事。

- Context Engineering 是管理模型上下文窗口里的信息，包括系统提示、工具定义、检索结果、历史消息和记忆。

- 如果要让 AI 更懂前端代码库，需要准备：架构说明、代码风格规范、常见组件示例、API 文档和错误处理约定。

- 复杂任务可以用 sub\-agent 或分块检索，把噪音隔离出去。

- 核心原则是：高信噪比、可复用、可评估。



### 日常如何使用 AI 编程工具？

要点：怎么用、什么时候用、如何验证。

- 用 Copilot / Claude Code / Cursor / ChatGPT 做脚手架、重构建议、测试补全和文档总结。

- 对复杂需求，会先让 AI 给方案，再自己审查设计是否合理。

- 对生成代码，会重点检查边界条件、可访问性、性能和安全。

- AI 更适合辅助思考和起草，不适合无审查直接合并。



### AI 生成代码不对时，怎么定位问题？

要点：把问题拆成 prompt、上下文、模型能力、代码实现四层。

- 先检查 prompt 是否明确，任务有没有歧义。

- 再看上下文是否缺少关键文件、接口或约束。

- 然后判断是不是模型本身能力不够。

- 最后再检查代码逻辑、类型、边界和测试结果。

- 把 AI 当成一个可能出错的同事，而不是权威。



### 如何用 AI 辅助完成前端功能的全流程？

要点：拆解、实现、验证、回看。

- 需求拆解：让 AI 列子任务和风险点。

- 方案设计：让 AI 生成多个实现方案，对比之后再做决策。

- 编码：让 AI 生成样板、类型、测试和边界处理。

- 测试：让 AI 协助补单测、列异常场景。

- 重构：让 AI 提醒可抽象的公共逻辑和可维护性问题。



### 什么是 Vibe Coding？它适合什么阶段？

要点：Vibe Coding 适合快速试错，不适合没有人工审查的正式生产交付。

- Vibe Coding 是更依赖 AI 快速生成、快速试错的开发方式。

- 它适合原型、demo、探索新想法、学习新 API。

- 它不适合高风险核心逻辑、复杂状态管理、支付/权限等关键路径。

- 真正上线的代码仍要经过 review、测试和安全检查。



### 如何保证 AI 生成 UI 组件的可访问性、样式一致性和边界情况？

- 需求结构化：技术栈、样式体系、UI状态

- 给 AI 明确约束：语义化要求、视觉规范、工程约束

- 分层实现：先布局、再细节，最后边界验证。可以考虑工具协助\(mcp\)

- 自动化验证：兼容性验证、屏幕自适应、多语言验证

- 人工审查：布局合理性、维护程度



### 如何保证 AI 工具下的代码质量、安全性和可维护性？

要点：AI 只是生产力工具，质量责任仍然在工程师。

- 给 AI 明确边界：不要改哪些文件、不要引入哪些依赖、必须满足哪些规范。

- 生成后必须有人 review，重点看安全、依赖、权限、数据泄露和错误处理。

- 对重复场景，要把规则写进模板、文档或项目约定里。

- 如果团队使用 AI 工具，需要推动建立统一规范，而不是每个人各写各的 prompt。
